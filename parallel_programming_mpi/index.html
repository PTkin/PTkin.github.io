并行程序设计(1) MPI

Posted on 2016-06-10

这学期上了一门《并行程序设计》，学习了并行程序设计的三种主流方式：分别是 MPI、OpenMP、Pthreads。于是打算写几篇笔记，对这门课学习的内容做一个简单的整理记录。

本文是第一篇，包括了并行程序设计的一些基础与 MPI 的基本使用方法，也包括了几个简单的算法的串行代码与使用 MPI 进行并行化之后的相应源码。

并行程序设计基础

并行化的两种思路

  1. 任务并行
  2. 数据并行

两种主要的并行系统

  1. 共享内存系统 (Pthreads, OpenMP)
  2. 分布式内存系统 (MPI)

名词区分

  1. 并行 (parallel)
  2. 并发 (cocurrent)
  3. 分布式 (distributed)
  4. 并发计算 (cocurrent computing)
  5. 分布式计算 (distributed computing)

冯诺依曼瓶颈 (Von Neumann Bottleneck)

  1. 指 CPU (包括算术逻辑单元 ALU 和 控制单元) 和主存的分离。
  2. 解决措施包括：
    1. 缓存
      • caching，一般指 CPU 中的 cache
      • hit 和 miss
      • 注意 write-through cache 和 write-back cache 的区别
      • cacha 映射方式有 全相联(full associative)、直接相联 (directed mapped)、n路组相联(n-way set associated)
      • 替换策略常用 LRU (least recently used)
    2. 虚拟内存
      • 交换空间 (swap space)
      • 页 (page)
      • 转译后备缓冲区 (Translation-Lookaside Buffer, TLB) ：页表，类似于 cache
      • page fault：类似于miss
      • 一般使用 write-back
    3. 指令级并行
      • Instruction-Level parallelism, ILP
      • 一般有两种实现：流水线、多发射
      • 多发射分为静态多发射和动态多发射，需要预测 (speculation) 技术

SIMD 系统

单指令多数据流 SIMD（single instruction, multiple data）

MIMD 系统

多指令多数据流 SIMD（multiple instruction, multiple data）

性能

  1. 几个符号：

    • 加速比 $ S $ (speedup)
    • 运行时间 $ T $ (time)
    • 核数 $ p $
    • 效率 $ E $ (efficiency)

  2. 加速比 $ S $

    • $ S = \frac{T_{串行}}{T_{并行}} $

  3. 效率 $ E $

    • $ E = \frac{S}{p} = \frac{T_{串行}}{p \cdot T_{并行}} $

Foster 方法

  1. 划分 (partitioning)

    识别出可以并行执行的任务，划分指令和数据

  2. 通信 (communication)

    确定上一步识别出来的任务之间需要的通信

  3. 凝聚或者聚合 (agglomeration or aggregation)

    合并需要合并的任务和通信等

  4. 分配 (mapping)

    将上一步聚合好的任务分配到进程/线程中（要求通信量尽量小，负载尽量均衡）

并行程序设计三种主流方式

  1. 消息传递接口 MPI

    (Message-Passing Interface)，适用于分布式内存系统。

  2. OpenMP

    适用于共享内存系统。

  3. POSIX 线程

    Pthreads (POSIX threads)，适用于共享内存系统。

MPI 基础

消息传递接口 MPI (Message-Passing Interface)，适用于分布式内存系统。

MPI 程序大体结构

函数

  • 通信子(communicator)

    一组可以互发消息的进程的集合

  • 函数

  • 消息匹配

    一般来说，除了满足 comm、tag、接收方和发送方的进程号分别对应上之外，还需要满足消息的数据类型 msg_type 相同，且接收方的缓冲区大小大于发送的消息大小，消息才可以正常匹配。

    注意这里 MPI 提供了两个变量 MPI_ANY_SOURCE 和 MPT_ANY_TAG，分别用于表示接收 任何发送方进程 的消息、接收 任何 tag 的消息。

编译与运行

  • 编译（以文件 mpi_hello.c 为例）

    mpicc -g -Wall -O2 -o mpi_hello mpi_hello.c
    • -g 表示允许使用调试器；
    • -Wall 表示显示警告
    • -O2 表示启动编译器优化（原理可网上搜索）
    • -o 后跟着输出的可执行文件名

  • 运行（以可执行文件 mpi_hello 为例）

    mpiexec -n <number of processes> ./mpi_hello
    • 或者使用命令 mpirun 表示快速执行，即不开启 mpdboot 守护进程
    • -n 后跟着执行的进程数

实验结论

以下：

  1. 分发数据的时间远远超过处理数据需要的时间。也就是说在现实的应用中，我们极有可能在计算能力上压力不大，反而在数据的传输、IO方面遭遇瓶颈。例如在矩阵向量乘法算法中，运算部分加速比超高、但总体加速比不高，原因在于分发数据的时间远大于处理数据的时间，会造成例如从 (1000+100) 优化到 (1000+10) 的尴尬情形。

  2. 线性加速比不可能达到，但是我们可以通过改换另一种并行化的思路，可以极大地影响并行化之后的加速比与效率。

  3. 在天河二号直接运行串行程序的耗时比使用单进程运行并行程序的耗时要高很多，猜测可能是由于天河二号中串行程序运行的默认机器主要用于分配任务，不需要高性能，因此使用了一台性能不高的机器。

  4. 在矩阵向量乘法中，有两次实验的效率超过了 100%，多次尝试之后依然可以得到类似的结果，猜测可能的原因如下：

    1. 代码有误（这个检查过多遍没有发现错误，但不排除）
    2. 当 n=1 的时候，天河二号会分配性能稍差的机器来跑，然后在多个性能稍高的核上进行并行处理的时候，有可能令效率超过 100%
    3. 天河二号的编译优化优化了一些不为人知的地方

补充：几个小坑

  1. ssh 连接天河二号报错 REMOTE HOST IDENTIFICATION HAS CHANGED!

    solution: 使用以下命令进入 known_hosts，删除天河二号 ip 对应的 ssh-rsa。

    vim ~/.ssh/known_hosts

  2. mpicc编译报错 error: expected an expression 之类的明明看上去代码没有问题却报了错

    solution: 天河二号默认支持的 C 语言版本比较老，我们在编译的时候使用配置 -std=C99 可以设置编译支持的 C 语言版本为 C99，即可解决问题。
