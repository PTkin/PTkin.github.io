折腾Hexo的搭建

Posted on 2016-05-13

很久之前就用 hexo 搭建过一个个人博客，但是觉得美化太麻烦，就弃了，后来断断续续在 CSDN 上写过几篇博客。最近越来越对其不满，觉得 CSDN 的 Markdown 渲染效果实在不能见人，而且毕竟有广告，所以决定重拾 Hexo 之路，顺便写下这篇博客记录下本次过程，顺便记录下搭建过程中遇到的几个坑。

Hexo 的搭建

Hexo 是一个快速、简洁且高效的博客框架，即使不会编程，也可能很轻松地上手。如果是程序猿/媛，即使像我这么笨，也只花了半个小时不到就搭建好了一个可以公网访问的个人博客。熟悉之后甚至不要十分钟就能搭起来 ~但是使用 Hexo 写博客需要掌握很简单实用的 Markdown 语法，如果你还不会的话建议先花 1 个小时学习熟悉一下再决定是否使用 Hexo 搭建个人博客吧 ~

搭建 hexo 的步骤在 Hexo官网 的文档里已经描述得非常详尽，这里不赘述。搭建成功之后默认是 landscape 主题，也可以根据自己的审美改换其他主题，可以点击 这里 查看我的另一篇文章《折腾Hexo的主题》。

下面上坑。

hexo server 不能本地访问

有以下几种可能：

  1. 忘记运行官方文档中的部分命令。例如：

    hexo init <folder>
cd <folder>
npm install

    建议先重新浏览官方文档查看是否按照其指示操作。

  2. 没有安装server模块（Hexo 3 起不自动安装）。我们可以手动安装：

    npm install hexo-server --save

  3. 本地主机端口被占用。我们运行命令时可以指定本地端口如下：

    hexo server -p 5000

    指定了端口 5000 。

  4. 检查代理设置，假如使用代理访问，自然出错，关闭即可。例如当时我就是挂着全局科学上网的 ss，结果 = =

ERROR Process failed: layout/_partials/.DS_Store

.DS_Store 是 OSX 保存文件夹的自定义属性的隐藏文件。

这里报错可能是因为我们在配置的过程中不小心改动了什么却没有清除缓存之类的错误，这个文件不是很重要，所以清除即可，清除之后会重新生成。

为什么会这样？简单来说就是由于 URL 中只允许使用规定范围内的字符，而中文恰恰不在其中=。=所以浏览器会对其做一定的编码处理，就变成了这个样子。

对于非 ASCII 字符，需要使用 ASCII 字符集的超集进行编码得到相应的字节，然后对每个字节执行百分号编码。对于 Unicode 字符，RFC 文档建议使用 UTF-8 对其进行编码得到相应的字节，然后对每个字节执行百分号编码。如 “中文” 使用 UTF-8 字符集得到的字节为 0xE4 0xB8 0xAD 0xE6 0x96 0x87，经过 Url 编码之后得到 %E4%B8%AD%E6%96%87。

如果要深究这个编码的问题可以参照 详解Javascript中的Url编码/解码 | 简单生活 — Kevin Yang的博客

那么问题来了

当我分享一个网址的时候，假如我分享的网址为 http://ptkin.github.io/2016/05/15/将Ubuntu安装到移动硬盘，那么别人(例如在微信中)点击该链接则会打开链接 http://ptkin.github.io/2016/05/15/，也就是说，后面的 将Ubuntu安装到移动硬盘 没有被识别为网址的一部分！此时打开的网址自然会 404 not found。

那么如果我直接发送编码之后的网址 http://ptkin.github.io/2016/05/15/%E5%B0%86Ubuntu%E5%AE%89%E8%A3%85%E5%88%B0%E7%A7%BB%E5%8A%A8%E7%A1%AC%E7%9B%98/ 呢？你惊喜地发现可以正常地打开，然后沮丧地发现

但是假如使用这种方法，那么生成的短链接根本无法看出原本的链接，所以还存在一个辨识度不高的问题。

方法二：

根据 Hexo 官方文档的 永久链接（Permalinks） 这一节，我们可以直接自定义每篇文章的 url。细节请直接查看官方文档，这里只记录我自己的做法。

  1. 打开 _config.yml Hexo 配置文件，修改 permalink 字段为自己想要的 url 格式（注意下面例子中的 en_title 也可以是别的，只要在配置文件和文章开头对应即可）

    permalink: :en_title/

  2. 在每篇文章的 Front-matter 中增加一行指定 en_title。例如

    en_title: install_ubuntu_to_ssd

  3. 生成静态博客之后发现该文章的 url 已经变成了

列表中插入代码块之后造成序号错乱

问题描述与解决方案类似 markdown的bug，博客预览与发布之后不一样。 - SegmentFault ：

因为生成页面时用的markdown引擎是PHP，然后预览用的是JS，是两套系统，所以在一些边边角角会有差异。

预览显示了符合期望的代码，但是实际上预览的引擎是有问题的，因为代码段和三个列表项是同一缩进，也就是层级是一样的，因此，代码段不应解析为第二个列表项的附属内容，而应该解析为独立的内容。然后第“三”个列表项被解析为新列表的开头（因为Markdown的数字列表，前面的数字会自动重排的）。所以预览引擎的处理其实是错误的。

但我实际发现，Markdown 语法本身并没有问题，或者说这个锅并不由 Markdown 背，而是应该怪罪于渲染引擎吧= =

至少当我如下写时：（注意下面块中为了防止块中块被解析成代码块所以用“点点点”代替了表示代码块的点点点）

1. one
2. two

    点点点bash
    my code1
    my code2
    my code3
    点点点

    hello

3. three

在我的 Macdown 编辑器中预览正常，在 CSDN 中发布也是正常，在 Hexo 中发布出现问题，会将我的 “hello” 页当成代码块，而 “3. three” 的序号则从1开始。

我只能修改成这样：

1. one
2. two

        my code1
        my code2
        my code3

    hello

3. three

COULD YOU SEE THE POINT ?

使用 4 个缩进来表示内嵌，再使用 4 个缩进来表示代码块。如此解决方案，虽然预览与发布的效果都正确，但是却无法指定语言，也是个缺陷。

修改代码渲染样式（Huxblog）

在我配置完 Huxblog 主题之后，发现它的代码高亮不尽人(我)意， 所以开始捣鼓如何修改 Huxblog 代码块的渲染样式。

Huxblog 的代码高亮配置文件是 themes/huxblog/source/css/syntax.styl ，但是 stylus 语言我一窍不通，于是我上网查是否有大牛配置好的 styl 文件可以直接使用。找了许久都没找到，于是在室友葛(前端大神)的帮助下，找到了 highlight.js，于是下载了特别喜欢的 monokai 在葛的帮助下配置成功了。步骤大致如下：

  1. 下载代码高亮渲染样式 css 文件 monokai-sublime.css，放入 themes/huxblog/source/css/

  2. 修改 themes/huxblog/layout/_partial/head.ejs 文件，在其中

    <%- css('css/syntax') %>

    后面增加一行代码变成

    <%- css('css/syntax') %>
<%- css('css/highlight.css') %>
